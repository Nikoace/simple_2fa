# 迁移到纯 React 架构说明

本文档用于指导从「`frontend_react + FastAPI + MCP`」逐步迁移到「纯 `frontend_react` 主线」的执行过程。

## 1. 迁移目标

- 将日常开发与运行默认收敛到 `frontend_react`。
- 将 FastAPI 与 MCP 能力降级为 **legacy/可选**，仅服务于兼容与过渡。
- 通过可审计的数据导出/导入流程保证账号密钥迁移可追溯。

## 2. 数据迁移策略

当前 legacy 后端使用 SQLite 持久化。迁移建议采用「导出 -> 校验 -> 导入」三段式：

1. **导出**
   - 从 SQLite 中导出账户基础字段（如 issuer、account_name、secret、period、digits、algorithm）。
   - 统一输出为 JSON（UTF-8），并附加导出时间与版本号。
2. **校验**
   - 校验 secret 是否为合法 Base32。
   - 校验 account_name 的唯一性冲突并给出冲突清单。
   - 抽样对比 TOTP 输出（旧系统 vs 新系统）确保一致。
3. **导入**
   - 前端导入 JSON 后进行二次校验。
   - 失败条目应可跳过并生成失败报告，避免全量回滚。

> 建议在迁移窗口内同时保留 legacy 后端只读能力，用于数据核对。

## 3. 风险清单

- **密钥泄露风险**：导出文件含敏感 secret，必须限制访问范围并设置过期销毁策略。
- **算法参数不一致**：若 period/digits/algorithm 映射错误，会导致动态码不一致。
- **重复账户覆盖**：同名账户导入时可能发生覆盖，需显式冲突处理策略。
- **兼容性风险**：部分历史数据可能存在非标准字段，需要灰度验证。

## 4. 回滚方案

若迁移后发现不可接受问题，按以下步骤回滚：

1. 暂停新数据写入（冻结导入入口）。
2. 恢复至迁移前的 SQLite 快照。
3. 切换运行方式到 `./dev_run.sh --with-backend`，恢复 legacy API 路径。
4. 使用迁移失败报告定位异常数据，修复后重新执行小批量迁移。

## 5. 分支策略建议

- `main`：保持 `frontend_react` 主线开发。
- `legacy/fastapi-mcp`（建议）：承载 FastAPI + MCP 的延续维护与补丁。
- 若需要长期并行，可将协议联动逻辑封装为独立包并在 CI 中做兼容验证。
